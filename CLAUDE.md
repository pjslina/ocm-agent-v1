# CLAUDE.md

用于减少大语言模型在编程中常见错误的行为准则。可根据项目实际情况与项目专属指引合并使用。

**权衡说明：** 这些准则倾向于稳健而非速度。处理琐碎任务时，请自行判断是否完全套用。

## 1. 编码前先思考

**不要假设。不要隐藏困惑。要把权衡摆在台面上。**

在动手实现之前：
- 明确陈述你的假设。如果不确定，就问。
- 如果存在多种可能的理解，请一并列出 —— 不要默默挑一种。
- 如果存在更简单的方案，请直说。必要时敢于反对。
- 如果有不清楚的地方，就停下来。指出哪里让你困惑，然后提问。

## 2. 简洁优先

**用最少的代码解决问题。不要做任何投机性的设计。**

- 不要添加需求之外的功能。
- 不要为只会被用一次的代码抽象接口。
- 不要引入未被要求的"灵活性"或"可配置性"。
- 不要为不可能发生的场景写错误处理。
- 如果你写了 200 行可以缩到 50 行，那就重写。

扪心自问："一位资深工程师看到这段代码会不会觉得过度复杂？" 如果会，那就再精简。

## 3. 手术刀式的改动

**只动你必须动的地方。只清理你自己留下的烂摊子。**

在编辑既有代码时：
- 不要"顺手优化"旁边的代码、注释或格式。
- 不要重构没坏的东西。
- 遵循既有风格，哪怕你个人风格不同。
- 如果你注意到无关的废弃代码，提一句即可 —— 不要擅自删除。

当你的改动造成孤立代码时：
- 删掉由你的改动导致不再使用的 import、变量、函数。
- 不要删除先前就存在的废弃代码，除非用户明确要求。

检验标准：每一行被改动的代码，都必须能直接追溯到用户的需求。

## 4. 目标驱动的执行

**先定义成功标准，再循环迭代直至验证通过。**

把任务转化为可验证的目标：
- "添加校验" → "为非法输入编写测试，然后让它们通过"
- "修这个 bug" → "先写一个能复现它的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，先简要列出计划：
```
1. [步骤] → 验证方式：[检查点]
2. [步骤] → 验证方式：[检查点]
3. [步骤] → 验证方式：[检查点]
```

清晰的成功标准能让你独立循环迭代。模糊的标准（"让它跑起来"）则会让你不断需要澄清。

---

**这些准则是否生效的判断依据：** diff 中不必要的改动变少了，因过度复杂而返工的情况变少了，并且澄清性的提问出现在动手实现之前，而不是在犯错之后。
